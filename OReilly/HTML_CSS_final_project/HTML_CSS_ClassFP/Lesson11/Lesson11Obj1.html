Hamsters: Who Knew There Were So Many Different Types?!

I had golden and teddy bears hamsters in my youth, but since then the types of hamsters have proliferated. I looked into getting a dwarf hamster some years ago, but shied away from them after I heard that they tend to get nervous and pee in your hands. (Something I could do without.)

Logic Behind the Table Structure

Why did I choose this data and what was my decision-making process for structuring the table to best manage that data? For instance, why did I pick rows or columns for the table headings? Did I create a separate cell for each item of data or did I combine some pieces of data, and if so, why?

The fact that there were so many different genus and species suggested that it would be best to use vertical table headers (with <th> in each <tr>) rather than horizontal ones (with all <th> in one <tr>). I put each category in a table row with the description of the particular species in a table cell (<td>) in the same row. So each row had a header (e.g., subfamily, genus, species) and possibly a description (e.g., Mongolian hamster).

Problem: This created a problem when I validated the file. All of my rows did not have the same number of columns. I received the error message: A table row was 2 columns wide and exceeded the column count established by the first row (1). Solution: Since some <tr> did not have both <th> and <td>, I added the attribute colspan="2" to the sole <th> or <td> tag in the <tr>.

Logic Behind the Table Styling and Its Impact on the HTML

I set up general styling with a rule for <body>. I found that I liked the table having default borders, which allowed the background color to come through in between the rows in the table, so I did not collapse the table borders.

Problem: I wanted the text in the header/the first column of the table to have a separate style from the second column. Solution: I set up a rule for <th> in the CSS.

Next problem: I didn't want the table to fill the entire browser window. Solution: I set up a rule for <td> and added a width that looked visually pleasing to the rules for <th> and <td>.

Next problem: I wanted to call out the rows that only had a header and no other cells, namely the subfamily and genus rows, and make them bold. Solution: By default, all <th> tags are bold. To make only the <th> with subfamily and genus bold, I added font-weight: normal; to the body rule; created two classes (subfamily and genus) for the <tr> tags that contained subfamily and genus th; and created a rule for these classes (tr.subfamily th, tr.genus th) that set the font-weight to bold.

Next problem: Some genus classes had extra text in parentheses that I didn't want bold. By creating a rule for tr.subfamily th, tr.genus th and setting the font-weight to bold, all text in these <th> tags had become bold. Solution: I added a <span> tag with a class of "tabheader" around these items (e.g., (golden hamsters)) and added a rule to the CSS (tr span.tabheader) to set the font-weight back to normal.

Next problem: I wanted to style the <source> content. Solution: I set up an id for "source" and then wrote various rules for the text as well as the link behavior. I also set up links elsewhere on the page to display only some of this behavior (e.g., links with the class "table").
